E-R Modeling from the Problem Statements

Objective: Identify entity sets, their attributes, and various relationships.

Represent the data model through an ER diagram.



Aim: To understand how to conceptually model a database using Entity-Relationship (ER) diagrams. By analyzing a given problem statement, we learn to identify entities, their attributes, and the relationships among them, which forms the foundation for building a well-structured database.

Introduction

Developing databases is a very important task to develop a system. Before going to form exact database tables and establishing relationships between them, we conceptually or logically can model our database using ER diagrams.

In this experiment we will learn how to find the entities, their attributes, and how the relationships between the entities can be established for a system.



Entity Relationship Model

The entity-relationship model represents a logical design of a database to be created. In the ER model, real-world objects (or concepts) are abstracted as entities, and different possible associations among them are modeled as relationships.

For this practical, the selected problem domain is an interactive quiz platform with users, quizzes, live sessions, responses, leaderboard updates, and analytics.



Problem Statement: Interactive Quiz Platform

The system is a real-time quiz application where registered users can create quizzes, host live sessions, and participate in quizzes using an 8-digit join code. A creator can prepare quizzes manually, by AI prompt, or by image scan. Each quiz contains multiple questions, and questions may contain options depending on question type.

When a host starts a session, participants join using the join code and submit responses in real time. The system calculates points, maintains ranks, and updates leaderboard data. It also stores detailed question-wise result analytics including correction rate and response distributions.

The data model must support user authentication, quiz lifecycle, session lifecycle, participant tracking, response tracking, and analytics generation.

Relationship Set with Cardinalities

1. User creates Quiz
- One User can create many Quiz records.
- Each Quiz belongs to exactly one User (creator).
- Cardinality: 1:N

2. Quiz contains Question
- One Quiz can contain many Questions.
- Each Question belongs to one Quiz.
- Cardinality: 1:N

3. Question has Option
- One Question can have many Options.
- Each Option belongs to one Question.
- Cardinality: 1:N

4. User hosts Session
- One User can host many Sessions.
- Each Session has exactly one host User.
- Cardinality: 1:N

5. Quiz is used to create Session
- One Quiz can be used in many Sessions.
- Each Session uses one Quiz.
- Cardinality: 1:N

6. Session has SessionState
- One Session has at most one SessionState.
- Each SessionState belongs to exactly one Session.
- Cardinality: 1:1

7. User participates in Session (through SessionParticipant)
- One User can participate in many Sessions.
- One Session has many participating Users.
- Implemented using associative entity SessionParticipant.
- Cardinality: M:N resolved into two 1:N relationships.

8. Participant submits Response
- One Session has many Responses.
- One Question can have many Responses.
- One User (participant) can submit many Responses across sessions/questions.
- Business constraint: one response per participant per question per session.

9. Response selects Option (through ResponseOption)
- One Response can map to many Options (multi-select).
- One Option can be selected by many Responses.
- Cardinality: M:N resolved by ResponseOption.

10. Session stores QuestionResult
- One Session has many QuestionResult entries.
- One Question can appear in many session-level QuestionResult entries.
- Unique pair (sessionId, questionId).



Keys and Constraints

Primary Keys
- Every entity uses id as primary key.

Candidate/Alternate Keys
- User.email is unique.
- Session.joinCode is unique.

Composite Unique Constraints
- SessionParticipant(sessionId, userId)
- Response(sessionId, questionId, participantId)
- ResponseOption(responseId, optionId)
- QuestionResult(sessionId, questionId)

Foreign Key Constraints
- Enforced across all child entities for referential integrity.



Generalization and Specialization

User is a generalized entity.

Specialized conceptual roles:
- Creator (User who creates quizzes and hosts sessions)
- Participant (User who joins sessions and submits responses)
- Admin (User with elevated role)

The specialization is represented through role and relationships rather than separate physical tables.



Weak / Dependent Entities

The following entities are existence-dependent on parent entities and can be treated as weak in conceptual understanding:

- SessionState depends on Session.
- SessionParticipant depends on Session and User.
- ResponseOption depends on Response and Option.
- QuestionResult depends on Session and Question.

How to Generate ER Diagram Image from Your Project

Option A: Generate directly from Prisma schema (recommended)

1) Install ERD generator package in repo root

npm install -D prisma-erd-generator

2) Add ERD generator block in Prisma schema (temporary or permanent)

generator erd {
	provider = "prisma-erd-generator"
	output   = "./ERD.svg"
}

3) Run generate command from repo root

npx prisma generate --schema packages/db/prisma/schema.prisma

4) Output file will be created at

packages/db/prisma/ERD.svg

You can then insert this image into your report.
