PRACTICAL 6

Use Case Modeling from the Problem Statement (Very Detailed)

OBJECTIVE

1. Identify every actor from a problem statement.
2. Identify every use case (services/functions of system).
3. Decide relationship types: association, include, extend, generalization.
4. Draw a proper UML use case diagram using correct symbols:
   - Actor -> stick figure (outside system)
   - Use case -> oval/circle (inside system)
   - System boundary -> rectangle
   - Relationships -> lines/arrows with labels

AIM

To create a complete UML use case model for an Interactive Quiz Platform with correct actor identification, use case identification, and relationship mapping.

INTRODUCTION

A UML use case diagram shows:
- Who uses the system (actors)
- What services the system provides (use cases)
- How use cases are connected (include, extend)
- How actors are specialized (generalization)

PROBLEM STATEMENT (INTERPRETED)

The platform allows creators to build quizzes, host live sessions, and control flow. Participants join via 8-digit code and answer in real time. The system computes score, updates leaderboard, stores analytics, and exports results.

STEP 5: ADD RELATIONSHIP ARROWS

A) Include (mandatory shared behavior)
Notation: dashed arrow with <<include>> from base to included use case.

1. Create Quiz -> Login
2. Edit Quiz -> Login
3. Delete Quiz -> Login
4. Launch Session -> Login
5. Start Session -> Launch Session
6. Submit Response -> Join Session via Join Code
7. Export Results (CSV) -> View Session Analytics
8. View Session Analytics -> End Session

B) Extend (optional behavior)
Notation: dashed arrow with <<extend>> from extension to base use case.

1. Rejoin Session -> Join Session via Join Code
2. Pause Session -> Start Session
3. Resume Session -> Pause Session
4. View Correct Answer -> Submit Response
5. Enable Adaptive Difficulty -> Configure Timer/Points/Difficulty
6. Generate Quiz from AI Prompt -> Create Quiz
7. Generate Quiz from Image Scan -> Create Quiz

C) Generalization
Notation: solid line with hollow triangle to parent actor.

1. Creator -> User
2. Participant -> User

STEP 6: EXACT DIAGRAM DRAWING ORDER

1. Draw one large rectangle (system boundary).
2. Write title: Interactive Quiz Platform.
3. Place actors outside rectangle.
4. Draw all use case ovals inside rectangle.
5. Connect actors to relevant use cases using solid lines.
6. Draw include arrows with <<include>> labels.
7. Draw extend arrows with <<extend>> labels.
8. Draw generalization arrows from Creator/Participant to User.
9. Check arrow directions.
10. Check readability and spacing.

STEP 7: QUALITY CHECKLIST

- Every use case name starts with a verb.
- No actor is placed inside rectangle.
- AI/OCR/Auth are actors, not use cases.
- Include and extend directions are correct.
- Diagram is legible and not cluttered.

DETAILED PLANTUML CODE

Save as usecase-diagram.puml

@startuml
left to right direction
skinparam packageStyle rectangle
skinparam shadowing false
skinparam linetype ortho
skinparam dpi 180

actor User
actor Creator
actor Participant
actor Admin
actor "AI Service" as AI
actor "OCR/Vision Service" as OCR
actor "Auth/Email Service" as AUTH

User <|-- Creator
User <|-- Participant
